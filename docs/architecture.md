### Rewritten Architecture Definition Document (ADD)

## 1. System Hierarchy & Granular Access Control (RBAC)

The platform operates on a robust multi-tenant architecture with a fine-grained, policy-based access control system, modeled after complex community platforms (like Discord or AWS IAM).

### 1.1 Core Multi-Tenant Entities
* **Global Users:** A central table of authenticated individuals (`id`, `email`, `global_name`, `password_hash`). A user requires only one account credential to access the platform.
* **Stores (Tenants):** The isolated business environments (e.g., independent C&F offices or vessel handling agencies).
* **Store_Memberships (The Pivot):** Connects a Global User to a Store. Crucially, this table holds the `role_id` and the user's specific `JSONB` custom data relevant only to that store.

### 1.2 The Discord-Style Permission Engine
Instead of broad roles, the system utilizes a **Resource:Action** permission matrix (e.g., `files:create`, `files:delete`, `requisitions:approve`, `settings:view`).

1.  **Store Roles:** A store contains customizable roles (`owner`, `manager`, `staff` generated by default). Each role contains a `JSONB` array of granted permissions.
    * *Note:* The `owner` role inherently bypasses all checks via a wildcard `["*"]` permission.
2.  **User-Level Overrides:** The `store_memberships` table contains an `overrides` JSONB column. This allows store admins to grant or revoke specific micro-permissions for an individual staff member without creating an entirely new role for them.
3.  **Permission Evaluation (The Cascade):** When a user accesses a store, the system calculates their Effective Permissions: `Base Role Permissions + User Additions - User Revocations`.
4.  **Execution & UI Rendering:** The Effective Permissions array is sent to the React frontend Context and encoded into the backend session token. React conditionally renders UI elements (hiding the "Delete File" button if `files:delete` is missing), while the API middleware actively rejects unauthorized HTTP requests.

## 2. Universal Dynamic Data Modeling (The Hybrid Schema)

The platform rejects rigid database structures in favor of a universal Hybrid Schema. Every major entity within a Store (Clients, Files, and Staff Memberships) is extensible via PostgreSQL `JSONB`.

### 2.1 The "Core + Custom" Entity Architecture
Every major entity is split into two distinct parts to balance performance with extreme flexibility:

1.  **Core Attributes (Relational DB Columns):** Hardcoded, universally required fields. These are never deleted or customized. They ensure the core application code, UI tables, and search engines always function.
    * *Client Core:* `id`, `store_id`, `name`.
    * *File Core:* `id`, `store_id`, `client_id`, `file_number`.
    * *Staff (Membership) Core:* `user_id`, `store_id`, `role_id`.
2.  **Custom Attributes (JSONB Column):** The dynamic, store-specific operational data.
    * *Client Custom:* `address`, `tax_id`, `industry`.
    * *File Custom:* `importer_name`, `vessel_arrival`, `container_count`.
    * *Staff Custom:* `emergency_contact`, `blood_type`, `department_name`.

### 2.2 The Global Metadata Engine
To manage this vast amount of dynamic data, the system utilizes a central `EntityFieldDefinitions` table.
* **Structure:** `id`, `store_id`, `entity_type` (Enum: Client, File, Staff), `field_name`, `field_type` (Enum: Text, Number, Date, Select), `options` (JSON array for dropdowns), and `is_required`.
* **Frontend Generation:** The React frontend fetches these definitions to dynamically generate data-entry forms. The backend validates incoming payloads against these definitions before saving them to the entity's JSONB column.

### 2.3 The Store Setup Wizard & Schema Seeding
When an Owner registers a new Store, they are routed through an interactive Setup Wizard.

1.  **Template Selection:** The wizard offers industry-specific preset configurations (e.g., "C&F Agency Preset", "Vessel Handling Preset").
2.  **Entity Configuration:** The owner is presented with optional fields for Clients, Files, and Staff.
3.  **Database Seeding:** Upon completion, the backend translates the wizard selections into records within the `EntityFieldDefinitions` table.
4.  **Lifecycle Management:** Post-setup, Managers can access the "Store Settings" to edit, hide, or create new custom fields. To protect historical data, deleting a custom field from the Definitions table performs a "soft delete" (hiding it from the UI) rather than wiping the historical JSONB data from the actual files or clients.

## 3. The Digital Imprest System (Treasury & Requisitions)

To handle the distribution of office funds for operational tasks, the system features a dedicated internal treasury workflow.

### 3.1 Virtual Wallets
Staff members do not have a static "balance" column. A user's wallet balance is dynamically calculated in real-time by summing all approved incoming funds and subtracting all recorded outgoing expenses.

### 3.2 The Requisition Workflow
1.  **Request:** A staff member submits a requisition (e.g., "Need $500 for customs transport on File #102"). The request enters a `Pending` state.
2.  **Review:** Managers or Owners view the request in the unified dashboard.
3.  **Approval:** The approver can accept the exact amount, alter the amount (e.g., approving only $400), or decline it.
4.  **Disbursement:** Upon approval, the system credits the staff member's virtual wallet, releasing the funds for use.

## 4. Job Costing Ledger (Expense Tracking)

This module tracks the actual capital leaving the virtual wallets and being applied to specific operations.

* **Data Structure:** Every expense is logged in a relational table linking the `user_id` (who spent it), `file_id` (what job it belongs to), `amount`, and a defined `category` (e.g., Port Fees, Labor, Transport).
* **Integrity:** This ledger is the absolute source of truth for the office's profit and loss metrics. It operates independently from the client-facing invoice to prevent financial data desync.

## 5. Dynamic Billing Engine (Invoicing)

The invoicing system generates "snapshot" documents for client billing. It maps internal job costs and conditional business logic into a clean, customizable PDF.

### 5.1 Template Architecture
* **Store Default Templates:** Managers design master invoice templates using a drag-and-drop UI.
* **Instance Overrides (Pro Forma):** When generating an invoice for a specific file, authorized users can temporarily edit the template descriptions, groupings, or formulas for that specific invoice instance without altering the master store template.
* **Layout Constraint:** To ensure visual stability, the generated document strictly adheres to a two-column layout: **Description** (Left, wide) and **Amount** (Right, narrow).

### 5.2 The Rules Engine & Block Types
The drag-and-drop builder is powered by `JSONLogic` to safely evaluate complex billing scenarios without executing raw code. Managers can utilize three block types:

1.  **Expense Blocks (Cost Pass-Through):** Automatically sums the actual expenses recorded against the file for selected categories.
    * *Validation Feature:* If a user attempts to generate an invoice but unbilled expenses exist in the Job Costing Ledger for that file, the UI throws a non-blocking warning: *"Unbilled expenses detected."*
2.  **Conditional Blocks (Rules Engine):** Applies flat fees based on dynamic `JSONB` file attributes.
    * *Example:* `IF (file.type == 'Export') THEN $5000 ELSE $6000`.
3.  **Formula Blocks (Margins/Taxes):** Performs mathematics on previously defined rows.

### 5.3 Circular Reference Prevention
The template builder enforces a **Top-Down Sequential Evaluation** methodology. When a user configures a Formula Block (e.g., Row 3), the UI dropdowns only permit them to select and reference variables from the rows physically positioned above it (Row 1 and Row 2). This structural constraint makes infinite calculation loops mathematically impossible.

## 6. Technology Stack Summary

The platform utilizes a decoupled, TypeScript-first architecture, leveraging a BaaS (Backend-as-a-Service) for infrastructure heavy-lifting while maintaining complete control over business logic through a custom backend API.

### 6.1 Infrastructure & Data Layer
* **Database:** PostgreSQL (Hosted via Supabase).
    * *Why:* Provides the essential relational integrity for multi-tenancy and the advanced `GIN` indexed `JSONB` columns required for the dynamic entity schema.
* **Authentication & Identity:** Supabase Auth.
    * *Why:* Handles secure user registration, session management, and JWT generation out-of-the-box, integrating seamlessly with PostgreSQL.

### 6.2 Backend (The Logic & Integration API)
* **Runtime:** Node.js.
* **Language:** TypeScript.
* **Framework:** Express.js or Fastify.
* **ORM / Query Builder:** Drizzle ORM.
* **Validation Engine:** Zod.
    * *Why:* Used to dynamically validate incoming API payloads against the store's custom metadata rules before database insertion.
* **Rules Engine:** `json-logic-js`.
    * *Why:* Safely parses and evaluates the conditional logic and formulas defined in the Dynamic Billing Engine templates.
* **Document Generation:** Puppeteer (or `@react-pdf/renderer`).
    * *Why:* Converts the aggregated JSON invoice data and HTML/React templates into downloadable, client-ready PDF documents.

### 6.3 Frontend (The Unified Dashboard)
* **Library:** React (with TypeScript).
* **Server State Management:** TanStack React Query.
    * *Why:* Handles caching, background fetching, and synchronization of dynamic database entities (Clients, Files, Requisitions) automatically, drastically reducing frontend boilerplate.
* **Client State Management:** Zustand.
    * *Why:* A lightweight, unopinionated store used for global UI states, such as managing the active user's session data, current store context, and the decoded RBAC permissions array.
* **Styling/UI:** Tailwind CSS (paired with a component library like shadcn/ui).

### 6.4 Observability, Testing, & Tooling
* **Logging:** Morgan (for HTTP request logging) and Winston (for structured, level-based application logging, tracking failed permission checks, and systemic errors).
* **Testing:** Vitest (for unit testing complex backend logic like the JSONLogic evaluator and Job Costing aggregations) and Supertest (for API endpoint integration testing).
